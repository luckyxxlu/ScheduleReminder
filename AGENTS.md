# ScheduleReminder

Windows 桌面作息提醒工具，推荐技术栈为 React + Rust + Tauri。

## 产品定位
- 这是一个以“提醒”而不是“任务管理”为核心的桌面应用。
- 目标用户是需要固定时间提醒、作息约束、轻量日历查看的个人用户。
- 产品应强调低打扰、低学习成本、明确反馈，不做复杂项目协作功能。

## 核心功能
- 定时提醒事件
- 宽容时间设置
- 日历展示
- 自定义提醒事件

## 推荐页面结构
- `今天`: 展示当前时间、下一条提醒、今日时间线、宽容中的提醒。
- `日历`: 默认月视图，支持查看每天的提醒分布和详情。
- `提醒`: 统一管理全部提醒模板，支持启用、暂停、编辑、复制。
- `设置`: 管理默认宽容时间、通知样式、开机自启、托盘、免打扰等。

## 关键交互约束
- 新建提醒必须足够快，常见场景应在一次表单内完成创建。
- 提醒弹窗必须提供明确动作：`完成`、`宽容 10 分钟`、`稍后提醒`、`跳过今天`、`打开详情`。
- “宽容时间”是产品特色，不等同于无限延期。超过宽容截止时间后，应将提醒标记为 `missed` 或升级提醒状态。
- 日历首先服务于“看见作息与提醒分布”，不是复杂排班系统。

## 数据与状态建议
建议至少维护以下实体：
- `ReminderTemplate`: 提醒模板
- `ReminderOccurrence`: 具体触发实例
- `ReminderActionLog`: 用户操作日志
- `AppSettings`: 应用设置

推荐状态流转：
- `pending -> grace -> completed/skipped/missed`

说明：
- Windows 通知派发视为一次事件记录，不作为长期持久化状态。
- 提醒内容既可以是自定义文本，也可以是固定功能事件，如关机。

## 架构建议
- 前端统一使用 TypeScript，不使用 JavaScript 文件作为业务实现代码。
- React 负责页面展示、表单交互、日历视图和设置页。
- Rust 负责调度器、本地数据库、托盘、系统通知和定时可靠性。
- 数据库存储改为 SQLite。
- 提醒调度不要只依赖前端定时器，应由 Rust 后台统一调度。

## UI 原则
- 保持 Windows 桌面端体验自然，不做过度移动端化布局。
- 视觉风格以系统原生感 + 温和卡片风格为主。
- 避免把首页做成普通任务列表，首页应突出“下一条提醒”和“今日节奏”。
- 颜色建议使用暖灰、墨绿、琥珀橙等稳重色系，避免通用蓝紫渐变模板。

## MVP 范围
- 单次提醒
- 每天/每周重复提醒
- 宽容时间
- 今日页
- 月历视图
- 自定义事件
- 系统托盘
- Windows 通知
- SQLite 持久化
- 开机自启

## 暂不优先实现
- 云同步
- 多端同步
- 团队协作
- AI 自动排程
- 复杂子任务体系

## 开发命令
```bash
# 前端 (React)
npm run dev

# 后端 (Rust/Tauri)
cargo tauri dev

# 构建
npm run tauri build
```

## 项目结构
- 前端源码: `src/`，优先使用 `*.ts` / `*.tsx`
- 后端源码: `src-tauri/`
- 入口: `src-tauri/src/main.rs`
- 产品文档: `docs/requirements.md`
- 技术设计文档: `docs/technical-design.md`
- 交互规格文档: `docs/ux-spec.md`
- 开发流程文档: `docs/development-workflow.md`

## 开发注意事项
- 当前目标平台是 Windows 桌面端。
- 前端新增页面、组件、状态管理、工具函数时统一使用 TypeScript，不新增 JavaScript 版本实现。
- 使用 Tauri 框架时，提醒、托盘和通知逻辑优先放在 Rust 侧实现。
- 后端脚本、Rust 模块和调度相关逻辑开发时，必须先编写测试用例，再进行功能开发。
- 后端测试用例应覆盖正常路径、边界条件、异常路径和状态流转，开发完成后必须先运行测试验证，再继续集成。
- 项目开发应以 `docs/requirements.md`、`docs/technical-design.md`、`docs/ux-spec.md` 和 `docs/development-workflow.md` 为基准，并按需求顺序推进。
- 每完成一个需求，必须遵循 `测试用例 -> 开发 -> 测试 -> 修复 -> 提交` 的固定流程。
- 使用 SQLite 时需明确数据库文件路径、初始化流程和 migration 策略，避免开发环境与运行环境不一致。
- 若实现重复规则、通知 API 或系统行为不确定，应先查官方文档再编码。
- 新增功能时优先保持模型和状态流转简单，避免为未来假设提前做过度设计。
