# ScheduleReminder Development Workflow

## 1. 文档目标

本文档定义项目开发顺序、需求拆分方式和单个需求的执行流程。项目开发必须以需求文档为基准，并遵循测试先行原则。

## 2. 基准文档

开发时应以以下文档为主：
- `docs/requirements.md`
- `docs/technical-design.md`
- `docs/ux-spec.md`

任何需求开发、测试补充和实现细节调整，应优先与以上文档保持一致。

## 3. 单个需求的固定执行流程

每开发一个需求，必须严格按照以下顺序执行：

1. 先写测试用例
2. 再开发功能
3. 运行测试
4. 修复失败项
5. 回归测试确认无问题
6. 确认通过后才可以提交

禁止顺序：
- 先写功能，后补测试
- 测试失败仍继续叠加需求
- 未验证通过就提交

## 4. 测试要求

后端需求必须覆盖：
- 正常路径
- 边界条件
- 异常路径
- 关键状态流转
- 数据库约束
- 系统动作风险场景

前端需求建议覆盖：
- 页面渲染
- 交互路径
- 表单校验
- 状态展示
- 弹窗按钮行为

## 5. 当前项目需求拆分顺序

以下需求顺序用于指导开发优先级。

## 5.1 基础工程与数据层

### 需求 1：项目基础框架初始化
目标：建立 Tauri + React + TypeScript + Rust 基础工程结构。

交付内容：
- 前端 TypeScript 工程结构
- Tauri 后端基础结构
- 基础路由与页面骨架

测试要求：
- Rust 基础模块测试框架可运行
- 前端基础渲染与路由可验证

### 需求 2：MySQL 初始化与 Migration
目标：建立数据库初始化、连接池和 migration 机制。

交付内容：
- `reminder_templates`
- `reminder_occurrences`
- `reminder_action_logs`
- `app_settings`

测试要求：
- migration 成功执行测试
- 重复执行兼容测试
- 唯一约束测试
- 数据库连接失败提示测试

## 5.2 提醒模板能力

### 需求 3：提醒模板 CRUD
目标：支持提醒模板创建、编辑、删除、复制、启停。

测试优先：
- create/update/delete/toggle/duplicate 测试
- 非法字段输入测试
- `event_type` 和 `event_payload_json` 合法性测试

### 需求 4：重复规则解析
目标：支持 `none`、`daily`、`workdays`、`weekly`。

测试优先：
- 正常规则解析测试
- 非法规则测试
- 边界日期测试

## 5.3 提醒实例与调度

### 需求 5：occurrence 生成
目标：根据模板生成未来提醒实例。

测试优先：
- 单次提醒生成测试
- 重复提醒生成测试
- 唯一约束防重测试

### 需求 6：调度器扫描与通知派发
目标：到时间后发送 Windows 通知，并记录 `notification_dispatched` 日志。

测试优先：
- `pending -> grace` 流转测试
- 通知派发日志测试
- 应用重启恢复测试

### 需求 7：宽容时间与延后机制
目标：支持 `grace_deadline_at` 和 `snoozed_until` 计算。

测试优先：
- 宽容 10 分钟测试
- 稍后提醒 5/10/15/30 分钟测试
- 超时进入 `missed` 测试

## 5.4 系统动作事件

### 需求 8：文本提醒事件
目标：完整跑通文本提醒流程。

测试优先：
- 提醒创建测试
- 通知派发测试
- 完成/跳过/延后测试

### 需求 9：系统动作事件 `shutdown`
目标：支持关机提醒，但必须经过明确确认。

测试优先：
- 关机提醒创建测试
- 取消测试
- 延后测试
- 确认后执行测试
- 未确认不得执行测试

## 5.5 前端页面能力

### 需求 10：今天页
目标：展示下一条提醒、今日时间线、宽容中的提醒。

测试优先：
- 页面渲染测试
- 状态颜色测试
- 快捷动作按钮测试

### 需求 11：提醒页
目标：管理提醒模板。

测试优先：
- 列表展示测试
- 启停按钮测试
- 编辑和复制入口测试

### 需求 12：日历页
目标：展示月视图和日期详情。

测试优先：
- 月视图渲染测试
- 日期点击测试
- 当日详情展示测试

### 需求 13：设置页
目标：管理默认宽容时间、托盘、主题、免打扰。

测试优先：
- 表单读写测试
- 保存状态测试
- 非法值校验测试

## 5.6 系统集成能力

### 需求 14：托盘
目标：支持最小化到托盘和托盘快捷菜单。

测试优先：
- 托盘菜单动作测试
- 窗口关闭行为测试

### 需求 15：开机自启与免打扰
目标：完成系统行为集成。

测试优先：
- 设置开关测试
- 免打扰时间段生效测试

## 6. 单个需求的执行模板

每次开发时都按下面模板推进：

### 需求名称
- 需求编号：
- 目标：
- 依赖：

### 第一步：测试用例
- 先列测试清单
- 先写测试文件
- 先跑测试确认当前失败方式符合预期

### 第二步：功能开发
- 只开发当前需求范围
- 不顺手混入下一个需求

### 第三步：测试验证
- 运行新增测试
- 运行相关回归测试

### 第四步：修复
- 修复失败用例
- 清理冗余逻辑
- 再次运行测试确认通过

### 第五步：提交
- 测试通过后再提交
- 提交信息应准确描述该需求的目的

## 7. 开发纪律

- 开发时必须按需求文档顺序推进，避免跳跃式实现。
- 若需求文档和实现方案冲突，应先更新文档，再编码。
- 若当前需求测试未通过，不进入下一个需求。
- 提交前必须确认测试通过。

## 8. 当前执行建议

推荐从以下顺序开始：

1. 需求 1：项目基础框架初始化
2. 需求 2：MySQL 初始化与 Migration
3. 需求 3：提醒模板 CRUD
4. 需求 4：重复规则解析
5. 需求 5：occurrence 生成

进入每个需求时，统一采用：

`测试用例 -> 开发 -> 测试 -> 修复 -> 提交`
