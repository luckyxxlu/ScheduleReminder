# ScheduleReminder Development Workflow

## 1. 文档目标

本文档定义项目开发顺序、需求拆分方式和单个需求的执行流程。项目开发必须以需求文档为基准，并遵循测试先行原则。

## 2. 基准文档

开发时应以以下文档为主：
- `docs/requirements.md`
- `docs/technical-design.md`
- `docs/ux-spec.md`

任何需求开发、测试补充和实现细节调整，应优先与以上文档保持一致。

## 3. 单个需求的固定执行流程

每开发一个需求，必须严格按照以下顺序执行：

1. 先写测试用例
2. 再开发功能
3. 运行测试
4. 修复失败项
5. 回归测试确认无问题
6. 确认通过后才可以提交

禁止顺序：
- 先写功能，后补测试
- 测试失败仍继续叠加需求
- 未验证通过就提交

## 4. 测试要求

后端需求必须覆盖：
- 正常路径
- 边界条件
- 异常路径
- 关键状态流转
- 数据库约束
- 系统动作风险场景

前端需求建议覆盖：
- 页面渲染
- 交互路径
- 表单校验
- 状态展示
- 弹窗按钮行为

新增强制要求：
- 不能只验证 mocked service 的成功路径，必须补充失败反馈测试。
- 创建、保存、切换状态等关键命令，必须验证前端参数传递与后端命令名一致。
- 页面布局改动若影响滚动区域，至少要验证主容器结构和滚动职责没有被破坏。
- 用户实际反馈优先级高于“测试已通过”，若真实运行与测试结果冲突，必须先修正测试再继续开发。
- 创建和保存流程必须验证可见反馈，至少覆盖成功提示、失败提示和表单校验提示。

## 4.1 真实可用性补充规则

- 左侧导航是全局结构，不应因页面内容过高而跟随主内容一起滚动。
- 当页面高度超出窗口时，只允许右侧内容区滚动；如移动端改为纵向布局，应明确按移动端单列规则处理。
- 今天页展示的“下一条提醒”必须与真实提醒实例一致，不能使用静态示例或占位时间线。
- 今天页的主操作对象必须与 Windows 通知语义一致：
  - 若提醒已进入 `grace`，页面主卡片应展示这条已触发通知的提醒。
  - 若提醒仍是 `pending`，页面应明确说明到点后才会发送 Windows 通知，且不应伪装成可立即处理。
- 日历页和提醒页中的创建/保存必须端到端生效，并在 UI 上提供明确成功或失败反馈，禁止出现“点了没反应”的状态。

## 5. 当前项目需求拆分顺序

以下需求顺序用于指导开发优先级。

## 5.1 基础工程与数据层

### 需求 1：项目基础框架初始化
目标：建立 Tauri + React + TypeScript + Rust 基础工程结构。

交付内容：
- 前端 TypeScript 工程结构
- Tauri 后端基础结构
- 基础路由与页面骨架

测试要求：
- Rust 基础模块测试框架可运行
- 前端基础渲染与路由可验证

### 需求 2：SQLite 初始化与 Migration
目标：建立数据库初始化、连接池和 migration 机制。

交付内容：
- `reminder_templates`
- `reminder_occurrences`
- `reminder_action_logs`
- `app_settings`

测试要求：
- migration 成功执行测试
- 重复执行兼容测试
- 唯一约束测试
- 数据库连接失败提示测试

## 5.2 提醒模板能力

### 需求 3：提醒模板 CRUD
目标：支持提醒模板创建、编辑、删除、复制、启停。

测试优先：
- create/update/delete/toggle/duplicate 测试
- 非法字段输入测试
- `event_type` 和 `event_payload_json` 合法性测试

### 需求 4：重复规则解析
目标：支持 `none`、`daily`、`workdays`、`weekly`。

测试优先：
- 正常规则解析测试
- 非法规则测试
- 边界日期测试

## 5.3 提醒实例与调度

### 需求 5：occurrence 生成
目标：根据模板生成未来提醒实例。

测试优先：
- 单次提醒生成测试
- 重复提醒生成测试
- 唯一约束防重测试

### 需求 6：调度器扫描与通知派发
目标：到时间后发送 Windows 通知，并记录 `notification_dispatched` 日志。

测试优先：
- `pending -> grace` 流转测试
- 通知派发日志测试
- 应用重启恢复测试

### 需求 7：宽容时间与延后机制
目标：支持 `grace_deadline_at` 和 `snoozed_until` 计算。

测试优先：
- 宽容 10 分钟测试
- 稍后提醒 5/10/15/30 分钟测试
- 超时进入 `missed` 测试

## 5.4 系统动作事件

### 需求 8：文本提醒事件
目标：完整跑通文本提醒流程。

测试优先：
- 提醒创建测试
- 通知派发测试
- 完成/跳过/延后测试

### 需求 9：系统动作事件 `shutdown`
目标：支持关机提醒，但必须经过明确确认。

测试优先：
- 关机提醒创建测试
- 取消测试
- 延后测试
- 确认后执行测试
- 未确认不得执行测试

## 5.5 前端页面能力

### 需求 10：今天页
目标：展示下一条提醒、今日时间线、宽容中的提醒。

测试优先：
- 页面渲染测试
- 状态颜色测试

### 需求 11：提醒页

测试优先：

### 需求 12：日历页

测试优先：

### 需求 13：设置页

测试优先：

## 5.6 系统集成能力

目标：支持最小化到托盘和托盘快捷菜单。

测试优先：

### 需求 15：开机自启与免打扰

## 6. 单个需求的执行模板

每次开发时都按下面模板推进：

### 需求名称
- 需求编号：
- 目标：
- 依赖：

### 第一步：测试用例
- 先列测试清单
- 先写测试文件
- 先跑测试确认当前失败方式符合预期

### 第二步：功能开发
- 只开发当前需求范围
- 不顺手混入下一个需求

### 第三步：测试验证
- 运行新增测试
- 运行相关回归测试

### 第四步：修复
- 修复失败用例
- 清理冗余逻辑
- 再次运行测试确认通过

### 第五步：提交
- 测试通过后再提交
- 提交信息应准确描述该需求的目的

## 7. 开发纪律

- 开发时必须按需求文档顺序推进，避免跳跃式实现。
- 若需求文档和实现方案冲突，应先更新文档，再编码。
- 若当前需求测试未通过，不进入下一个需求。
- 提交前必须确认测试通过。

## 8. 当前执行建议

推荐从以下顺序开始：

1. 需求 1：项目基础框架初始化
2. 需求 2：SQLite 初始化与 Migration
3. 需求 3：提醒模板 CRUD
4. 需求 4：重复规则解析
5. 需求 5：occurrence 生成

进入每个需求时，统一采用：

`测试用例 -> 开发 -> 测试 -> 修复 -> 提交`

## 9. Git 提交与推送注意事项

### 9.1 提交前提
- 只有在相关测试通过后，才允许执行提交。
- 提交后默认继续推送到远端，不把“提交”和“推送”拆成两次单独动作。
- 若提交时 Git 提示缺少 `user.name` / `user.email`，优先使用临时环境变量完成本次提交，不修改全局 Git 配置。

PowerShell 示例：

```powershell
$env:GIT_AUTHOR_NAME='你的名字'
$env:GIT_AUTHOR_EMAIL='your-email@example.com'
$env:GIT_COMMITTER_NAME='你的名字'
$env:GIT_COMMITTER_EMAIL='your-email@example.com'
git commit -m "message"
```

### 9.2 Windows 下推送 GitHub 的网络注意事项
- 当前仓库远端使用的是 HTTPS，不要求必须配置 SSH 才能推送。
- 浏览器可以打开 GitHub，不代表命令行里的 `git`、`curl`、`git ls-remote` 一定能直连 GitHub。
- 若出现以下错误，优先怀疑“浏览器走了系统代理，但 Git 没走”：
  - `Recv failure: Connection was reset`
  - `Failed to connect to github.com port 443`
  - `HTTP 408`

### 9.3 排查顺序
1. 先确认远端地址是否正确：`git remote -v`
2. 先测试远端读取是否通：`git ls-remote origin`
3. 若失败，再检查 Windows 系统代理：

```powershell
Get-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Internet Settings" |
  Select-Object ProxyEnable, ProxyServer, AutoConfigURL, AutoDetect
```

### 9.4 发现系统代理已开启时的处理方式
- 若 `ProxyEnable = 1` 且 `ProxyServer` 有值，优先让当前终端会话临时复用该代理。
- 不要为了单次推送去修改全局 Git 配置。

PowerShell 示例：

```powershell
$env:HTTP_PROXY='http://127.0.0.1:代理端口'
$env:HTTPS_PROXY='http://127.0.0.1:代理端口'
git ls-remote origin
git push origin main
```

说明：
- `代理端口` 以系统当前代理为准，不要写死到文档之外的固定值。
- 推送前先跑 `git ls-remote origin`，确认链路通了再执行 `git push`。
- 只在当前命令会话里注入代理环境变量，避免影响其他仓库或系统环境。

### 9.5 当前项目的已验证经验
- 本项目曾出现“浏览器可访问 GitHub，但 Git 无法推送”的情况。
- 根因是 Windows 用户代理已开启，但 Git/终端默认未复用该代理。
- 复用系统代理到 `HTTP_PROXY` / `HTTPS_PROXY` 后，`git ls-remote origin` 与 `git push origin main` 均恢复正常。
