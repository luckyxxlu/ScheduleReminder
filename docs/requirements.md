# ScheduleReminder Requirements

## 1. 文档目标

本文档用于定义 ScheduleReminder 的首版产品需求、业务边界、核心交互和落地范围，作为设计、开发和测试的共同依据。

## 2. 产品概述

### 2.1 产品名称
ScheduleReminder

### 2.2 产品定位
一个面向 Windows 桌面场景的轻量时间提醒器，用于帮助用户管理固定作息、规律提醒和自定义时间事件。

### 2.3 核心价值
- 让固定时间提醒更容易创建和执行。
- 通过“宽容时间”机制降低打断感，同时避免无上限拖延。
- 用日历可视化展示提醒分布和作息节奏。
- 支持用户快速创建一次性或重复的自定义提醒事件。

### 2.4 非目标
- 不做复杂团队协作。
- 不做完整项目管理工具。
- 不在首版引入云同步和多端同步。

## 3. 目标用户

### 3.1 典型用户画像
- 需要规律作息提醒的上班族。
- 需要固定时间学习或休息提醒的学生。
- 希望通过桌面工具管理睡觉、喝水、拉伸、会议前准备等固定事件的个人用户。

### 3.2 用户核心诉求
- 提醒稳定可靠，不容易错过。
- 设置过程简单，不需要复杂学习。
- 被提醒时能快速处理，而不是被迫进入复杂页面。
- 能看到今天和本周的提醒分布。

## 4. 功能范围

### 4.1 MVP 功能
- 单次提醒
- 每天/每周重复提醒
- 宽容时间设置
- 今日页
- 月历视图
- 自定义事件创建与编辑
- 托盘常驻
- Windows 系统通知
- MySQL 持久化
- 开机自启

### 4.2 后续版本功能
- 周视图和列表视图
- 拖拽改期
- 标签和颜色分类
- 免打扰时段
- 统计页
- 数据导入导出

## 5. 业务概念定义

### 5.1 提醒模板 ReminderTemplate
用户定义的一条提醒规则，描述提醒标题、重复方式、默认宽容时间和通知配置。

### 5.2 提醒实例 ReminderOccurrence
根据提醒模板生成的一次具体触发记录，带有计划触发时间、宽容截止时间和处理状态。

### 5.3 宽容时间 Grace Period
从提醒触发时间开始，到宽容截止时间结束的一段可接受延后处理窗口。

### 5.4 自定义事件
由用户自由创建的提醒，不限定于作息类，可用于会议、喝水、拉伸、学习、睡觉等。

### 5.5 事件类型
首版提醒事件分为两类：
- 文本提醒事件：以自定义文本内容为主，用于常规提醒。
- 系统动作事件：以固定功能为主，例如关机。

系统动作事件在首版中应受到更严格约束：
- 必须由用户显式创建。
- 触发时应通过 Windows 系统通知或应用弹窗明确提示即将执行的动作。
- 应提供取消或延后入口，避免误触发造成破坏性后果。

## 6. 功能需求

### 6.1 提醒创建
用户可以创建一条提醒，至少包含以下字段：
- 标题
- 分类
- 日期
- 时间
- 是否重复
- 宽容时间
- 提醒方式
- 备注

重复规则至少支持：
- 不重复
- 每天
- 工作日
- 每周指定天

验收要求：
- 用户应在单个创建流程中完成提醒配置。
- 常见场景下不超过一次主表单跳转。

### 6.2 提醒触发
当系统时间到达提醒计划时间时，应用应触发提醒。

提醒方式至少包括：
- 系统通知
- 应用内弹窗
- 可选声音提示

提醒内容支持：
- 自定义文本提醒
- 固定功能事件提醒，例如关机提醒

验收要求：
- 已启用提醒必须在计划时间附近稳定触发。
- 应用从后台或托盘状态下仍可正常通知。
- 通知触发应记录为一次事件日志，而不是新增一个长期状态。

### 6.3 宽容时间
当提醒被触发后，系统进入宽容期，直到 `scheduled_at + grace_minutes`。

宽容期内用户可执行：
- 完成
- 稍后提醒
- 宽容 10 分钟
- 跳过今天

动作定义：
- `宽容 10 分钟`：固定将当前提醒延后 10 分钟。
- `稍后提醒`：用户从预设选项中选择延后时长，首版建议支持 5/10/15/30 分钟。

超过宽容截止时间仍未处理时：
- 提醒状态更新为 `missed`
- 在今天页和提醒列表中高亮显示
- 可按设置决定是否进行二次提醒

### 6.4 今日页
今日页用于展示用户当天最重要的提醒信息。

页面应包含：
- 当前时间
- 下一条提醒
- 今日时间线
- 正在宽容中的提醒
- 今日已完成数量

验收要求：
- 用户打开应用后无需切换页面即可理解当天状态。

### 6.5 日历展示
首版默认提供月视图。

页面能力包括：
- 显示每天的提醒数量
- 通过颜色区分不同分类
- 点击日期查看当天提醒详情
- 从日历进入提醒详情或编辑

验收要求：
- 用户可快速判断某天是否有提醒和提醒密度。

### 6.6 提醒管理页
提供全部提醒模板的统一管理能力。

应支持：
- 查看全部提醒
- 启用/暂停提醒
- 编辑提醒
- 复制提醒
- 按状态筛选

### 6.7 设置页
首版设置应包含：
- 默认宽容时间
- 通知声音开关
- 开机自启
- 托盘常驻
- 免打扰时段
- 主题模式

## 7. 页面结构

### 7.1 今天
目标：让用户一眼看到当前节奏和最近提醒。

建议布局：
- 顶部：当前时间、今日状态、下一条提醒倒计时
- 中部：今日时间线、即将到来、宽容中的提醒
- 底部：今日完成数、快捷创建入口

### 7.2 日历
目标：查看提醒在时间维度上的分布。

建议布局：
- 左侧或顶部为月历主体
- 右侧为选中日期详情面板
- 支持快速跳转今天

### 7.3 提醒
目标：管理提醒规则，而不是展示单日执行情况。

建议布局：
- 顶部筛选和搜索
- 列表按下一次触发时间排序
- 每项提供启用、暂停、编辑、复制操作

### 7.4 设置
目标：管理系统行为和偏好。

## 8. 关键交互要求

### 8.1 提醒弹窗
提醒弹窗必须提供以下按钮：
- 完成
- 宽容 10 分钟
- 稍后提醒
- 跳过今天
- 打开详情

弹窗信息应包含：
- 标题
- 当前时间
- 原定提醒时间
- 宽容截止时间

若当前提醒为系统动作事件，例如关机：
- 弹窗必须明确展示即将执行的动作类型。
- 弹窗应优先提供取消、延后和打开详情入口。
- 未经用户确认，不应直接执行高风险系统动作。

### 8.2 快速创建
应用应支持从主页面或托盘菜单快速进入创建提醒流程。

### 8.3 空状态
当今日无提醒时，应展示鼓励性空状态和快捷创建入口，而不是空白页面。

## 10. 状态流转

推荐状态：
- `pending`
- `grace`
- `completed`
- `skipped`
- `missed`

状态说明：
- `pending`: 已生成但未触发
- `grace`: 正处于宽容期
- `completed`: 用户已完成
- `skipped`: 用户主动跳过
- `missed`: 宽容期结束仍未处理

补充说明：
- Windows 通知发送本身属于一次派发事件，应记录在操作日志中，例如 `notification_dispatched`，不单独作为长期状态保存。

## 11. 非功能需求

### 11.1 平台
- 当前仅支持 Windows 桌面。

### 11.2 技术架构
- 前端：React + TypeScript
- 后端：Rust
- 桌面框架：Tauri
- 数据存储：MySQL

### 11.3 可靠性
- 提醒调度不应只依赖前端定时器。
- 应用重新启动后应能恢复提醒状态。
- 系统休眠恢复后应重新计算错过或即将到来的提醒。
- 应用启动时应验证 MySQL 连接可用性，并对连接失败给出明确提示。

### 11.4 性能
- 常驻后台时资源占用应保持轻量。
- 主界面应快速打开，常用视图避免明显卡顿。

### 11.5 可用性
- 常见操作不超过 3 次点击。
- 文案应直接、明确、避免专业术语。

### 11.6 前端技术约束
- 前端开发统一使用 TypeScript。
- 新增页面、组件、hooks、状态管理和工具函数使用 `*.ts` 或 `*.tsx`。
- 不使用 JavaScript 作为前端业务实现语言。

### 11.7 后端开发测试约束
- 后端脚本、调度器、数据库访问层和系统动作执行逻辑必须先写测试用例，再进行功能开发。
- 测试用例应覆盖正常路径、边界条件、异常路径和关键状态流转。
- 功能开发完成后，必须使用已编写的测试用例进行验证，通过后才能进入联调。
- 涉及高风险系统动作，例如关机，必须补充取消、延后、误触防护和恢复场景测试。

## 12. UI/视觉原则

- 风格定位为系统原生感 + 温和卡片风格。
- 首页以信息层级清晰为主，不做复杂装饰。
- 避免通用蓝紫渐变模板。
- 推荐色系：暖灰、墨绿、琥珀橙。
- 深色模式应兼顾夜间使用舒适度。

## 13. 风险与注意事项

- Windows 通知、托盘、开机自启等行为具有平台差异，需要优先验证实际可行性。
- 重复规则和宽容时间会带来提醒实例生成与状态同步复杂度，应在首版保持规则简单。
- 不应把宽容时间设计成无限拖延机制，否则会削弱产品价值。
- MySQL 引入额外运行环境依赖，需要明确是连接本地 MySQL 服务还是远程 MySQL 实例。
- 数据库连接、凭据管理和初始化失败提示需要作为首版可用性的一部分考虑。

## 15. 验收标准摘要

- 用户可以创建单次和重复提醒。
- 提醒到时可以通过系统通知或应用弹窗稳定出现。
- 用户可在宽容期内完成、延后或跳过提醒。
- 月历可以展示每天提醒分布。
- 应用关闭主窗口后可托盘常驻。
- 重启应用后提醒数据和状态能够恢复。
